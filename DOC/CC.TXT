                            Introduction

   Throughout the entire development of the optimized SOLID C compiler, the goal was to create
comprehensive documentation that would describe its work in the system itself and programming
assistance for it. To achieve this goal, a detailed description of individual tools was created. Thus, when
developing a tool for automatic register allocation, it was possible to remove from the code
effectively unneeded object code. As a result, the code remained on a high-level language (similar to
ESTEX itself) with a system that works under strict restrictions on memory and real-time constraints. At the same
time, it was necessary to preserve the required programming for all assemblies.

   The main goal of this documentation - to help the user understand and apply this compiler in the framework of the ESTEX
operating system and to develop proper use of this compiler.

   SOLID C can be represented as a two-stage compiler language, transforming source C-text and
programs into assembly language. The compiler consists of two files "cc1.exe" and "cc2.exe", which
perform the corresponding stages and phases of transformation.
   In the first phase, the input C-text is transformed into an intermediate form (C-form),
stored in a file with the extension ".tmc" (by convention). In the first phase,
such diagnostics of errors are provided and formatted messages are output.
   In the second phase, the intermediate C-form is transformed into assembly code,
stored in a file with the extension ".asm" (by convention).
   It should be noted that for a significant number of constructs, the optimization is performed quite well, but the time spent
on the optimization by the C-program is 2 to several times longer compared to non-optimized (non-optimized) C-compilers.


                     Description of Compiler Stages

   The first stage (cc1.exe) performs the first phase of compilation. It reads
the source C-text, checks it for correctness of syntax and semantics. As a result of the first stage, an intermediate
file with C-form is created, i.e., C-text. The standard extension of this file, containing
C-form, is ".tmc".
   The command line for the first stage has the following format:

       CC1 [options] source_file

   The source file may contain uppercase letters and numbers, and the maximum length
of the line must not exceed 80 characters. Typically, the source C-file uses the extension ".c". If the extension
of the source file is ".c", then you do not need to specify it.

                  Options for Compiling Object Code

 -c       Unlike the standard CC mode, the SOLID C compiler
          processes static (declared) identifiers.
          This option enables the static identifier processing mode.

 -dNAME   Define a preprocessor constant in the form "NAME". The option value can be
          "NAME" may not be checked.

 -e       Enable preprocessing, output messages to error file with extension
          ".err". This is created always from the current name of the source
          ".c" file.

 -jN      Set the maximum number of error messages for compilation. Compilation is
          aborted if the number of errors equals or exceeds "N". The option value
          with the number "N" may not be checked. Example: -j30 (abort compilation
          after 30 errors).

 -k       Enable compatibility mode with the standard CC language (K&R standard).
          It allows you to use the old form of function parameters.
          This option must be used when compiling "old" source code.

 -m       Enable output of diagnostic information in table form.

 -oNAME   Specify the name for the output file in C-form. The maximum length
          of the path (folder\subfolder\file) must be no more than
          80 characters. The option value with the string "NAME" may not be checked.

 -rP:S:H  Set the relative sizes of parts of the working symbol table:
          P - pool, S - symbol table, H - hash-table.
          By default, the ratio 13:6:4 is used. If there is overflow
          of any part of the table, use it for another
          appropriate ratio of the sizes of the working parts of the table.
          The option value with the string "P:S:H" may not be checked.

 -x       Enable extended compatibility with MSX-C compiler. At the same time,
          the extension of the C-form file is changed from ".tmc" to ".tco".


                           Assembler

   The assembler (cc2.exe) performs the second phase of compilation. It reads
a file in C-form and creates the output file in assembly language.
   The command line for the second phase has the following format:

        CC2 [options] source_file

If the extension of the source file in C-form is not specified, then the
standard extension ".tmc" is used. If the extension is not specified for the output
file - the extension ".asm" is used. The maximum length of the string
"folder\subfolder\source_file" should be no more than 64 characters.
If you want to use a file for output that does not exist, then by default
it will be created in the current directory of the source C-form file.

                       Assembler Options

 -k      Do not delete the source C-form file (.tmc) after successful completion
         of the assembly.

 -oNAME  Specify the name for the output asm-file. The maximum length
         of the path (folder\subfolder\file) should not exceed
         64 characters. The option value with the string "NAME" may not be checked.

 -rN     Reserve memory for the symbol table of the compilation with a size
         of N bytes (N - decimal format). The option value with the number "N"
         may not be checked. Example: -r2000 (reserve symbol table with 2000 bytes).

 -uX     Define a prefix-character "X" for labels in asm-code. By default,
         the character "_" is used. Example: -u@  (use the prefix-character
         "@" for labels in the output asm-code).

 -q      "Quiet" mode. The compiler suppresses output of "---" and "+++"
         messages to the screen during execution.




              Compatibility of SOLID C Compiler for the ESTEX System


   It was said earlier that the entire development of the new C-compiler was carried out
for compatibility, and therefore the entire approach was to (identical assembly-level language scheme)
develop programs that allow the use of strict requirements of real-time and memory constraints. In terms of
compatibility with the standard, exceptional attention was paid to this issue. Moreover, there should be
clarification of the programming in SOLID C.

   The SOLID C compiler can work in three different modes. First, one can "comply" with the behavior of the standard
C compiler, and second - help ensure compatibility with the standard C. In this mode, automatic
type conversion is performed exactly as in standard C. The next difference between this mode and "compliance"
mode is that in the compliance mode, automatic type conversion of formats for the required
types "char" and "int" is not performed. By default, the compiler works in "compliance"
mode. For switch mode from operation, use the option "-k" in the first stage of compilation.


                   Compatible Mode of Compilation

   In this mode are supported for two types of format: integer constant (intconst)
and character constant (charconst). Typically specified formats are used exactly
in the corresponding formats "int" and "char", except in cases where other type
designations and representations are preserved and converted into corresponding,
specified formats mismatch for different.
  For simplification, in this mode, automatic type conversion formats for the required
types "char" to "int" are not performed. The following agreements with type conversion apply:

   unsigned  + ?  ->  unsigned
   int       + ?  ->  int
   char      + ?  ->  char
   intconst  + ?  ->  intconst
   charconst + ?  ->  charconst


              Compatibility Mode with Standard C Language

   Perform all arithmetic type conversions, consistent with the implementation
of the standard C language. The main difference consists in the fact that SOLID C
implements the type "char" for all assigned values. For this mode, the following
rules of type conversion apply:

   char            ->  int
   unsigned + ?    ->  unsigned
   int      + int  ->  int



                        Recursive Functions

   It is important that functions of the standard C language are recursive. In the framework of the SOLID C
compiler there may be both recursive and non-recursive functions. For distinguishing
functions there must be a type specifier and pre-reserving the following keyword:
"recursive" - recursive and "nonrec" - non-recursive. These keywords
are placed in the declaration of the function before the keyword and curly
braces to perform. For clarification, the following example:

nonrec main()
{
   printf("Hello, world!\n");
}

recursive
int factorial(n)
int n;
{
   return(n > 1) ? n*factorial(n-1) : 1;
}

int max(a,b)
int a,b
{
   return(a > b) ? a : b;
}

   In the given example, two functions are defined: non-recursive function main() and
recursive function factorial(). The function max() is deliberately shown without mode
designation, therefore it will be executed as it is compiled by default. Typically,
by default, the mode of execution of functions without recursive designation is used.
For clarity, the designation can be changed using the following preprocessor directives:

     #pragma nonrec
     #pragma recursive

   The first designation by default sets the mode of execution of functions as
non-recursive. The second directive defines the nature for functions that
are recursive (if it is not explicitly specified).
For example, if no designation #pragma is used, the function max() will be executed
according to the subsequent default mode of "recursive/non-recursive".
The keyword "nonrec" allows, if necessary, to force the function to be non-recursive.
At the same keywords, the compiler performs segmentation of generated object code,
which becomes and occurs in a different way than in other compilers. In the development
of some programs, a large number of functions may be non-recursive, therefore at the
beginning of the source text should follow the directive designation: #pragma nonrec


                   Functions with Variable Parameters

   The SOLID C compiler handles functions of two types: with variable number
of parameters and with fixed parameters. In functions of the first type the number and
description of parameters may be unlimited, then for in functions of the second type for
the same designation method. For example, functions printf, scanf and many other standard
functions have a variable number of parameters. In the SOLID C compiler functions are
designated with both types separately and using specialized declarations as follows:

     In function declarations:
        declarator
           (declarator)
           *declarator
           declarator()
           declarator(.)
           declarator [constant expression]

     In function pointers:
        pointer declarator
           *pointer declarator
           pointer declarator()
           pointer declarator(.)
           pointer declarator [constant expression]

   The designations "declarator(.)" and "pointer declarator(.)" indicate special
for designation of functions with variable parameters. The designations "declarator()"
and "pointer declarator()" characterize functions with fixed parameters.
This method of designation of functions with variable parameters is used in the
declarations of type cast, sizeof. For example, in the following designation of function func and fp
is defined, that function with variable parameters:

     int func(.), (*fp)(.);

Before applying a function with variable parameters, we must often expand the parameter
list in different ways. In practice, standard functions are functions with variable
parameters, for example:

     printf, fprintf, sprintf, scanf, fscanf, sscanf

The designation of these functions is contained in the standard file "stdio.h".
The call of a function with variable parameters is completely consistent with the call of a function
with fixed parameters.
   For functions that do not provide clear notation at the point of labor (lack
of method of passage and parameters). Possibility of designation is only for the same
function with variable parameters. Different functions, by the way, have optional
parameters.
   The definition of a function with variable parameters should be in the same
beginning, before the body. Then appropriate type designation of function func,
having function with variable parameters:

int func(.);       /* This designation must be at the beginning */

int func(nargs, args)
int nargs, args;
{
  ...
}

   In the process of calling a function with variable parameters, several actual
parameters are passed as two - receive call of the assignment. The value of nargs
parameter contains the number of actual parameters - fixed beginning. Options
for each parameter, how many actual parameters are really defined in the function
with variable parameters on the calling side function. This limitation is determined
by the fact that only the number can be defined parameters. At the same time, install
type of the result for all parameters specified.
   In order to organize the access and specify the parameters, we need to calculate
offset from the defined parameters for call of the function of parameters. List of parameters
(if not known) is used to search the array args. In other words, the value
of the first of the parameters args is forwarded direct offset args. Then the next
parameters are received from the array, shifted in the direction of increasing address. The offset is set
sequentially as follows:

     Offset of 2nd parameter = offset of args + 2
     Offset of 3rd parameter = offset of args + 4

In the following table is given the correspondence of parameters to their offsets:
________________________________________________

  Parameter Number          Offset
________________________________________________

       1             Offset of beginning of args array
       2             Offset of beginning + 2
       3             Offset of beginning + 4
      ...
       n             Offset of beginning + (n-1)*2
________________________________________________

   Careful correspondence of offsets to sequential parameters, given
in the table, allows to organize access to these parameters. For example, for that
to read from the array the second and third parameters of type "char", you need to include
in the program:

     (char)args         - reference to the first parameter of type "char"
     (char)+(&args+1)   - reference to the second parameter of type "char"

Note: if in the given example args has type "int", then according
the designation, the value of &args is converted to type "int". Then following from the next
document, the unit should be calculated according to the module &args+1, and the next unit "1" is
converted in correspondence with the table of relative offset from the pointer unit:
for that it is the same as for the beginning (except, the unit is defined for beginning
in the integer expression sizeof(int*)). In the appropriate case it is "offset of beginning
of array of parameters at some zero". Subsequently, the offset can be used to organize access
to parameters of other types.
   In the following sections, given examples of two functions with variable parameters:
function sum returns the value of the sum of variable number of "int" parameters.
Function cmax returns the maximum value from a range of "int" parameters and TINY (char),
the number of the subsequent - variable.

#pragma nonrec
typedef char   TINY;

int sum(.);
TINY cmax(.);

int sum(nargs, args)
int nargs, args
{
    int i, *p;
    for(i=0, p=&args; nargs --;)
          i += *p++;
    return i;
}

TINY cmax(nargs, args)
int nargs;
TINY args;
{
    int *p, max;
    for(max=0, p=(int*)&args; nargs--; p++)
        if(max < (TINY)*p)
            max = (TINY)*p;
    return max;
}


                Conversion of Logical Arithmetic

   In order to achieve effective object code, the SOLID C compiler performs
conversion of logical arithmetic. The essence of this optimization is that conversion
of variables, returned type "char", is performed not only in whole, but also in parts.
Since the processor Z80 and its instruction format make addition very efficient,
but others less so. Logical arithmetic operations are especially effective
for designation of specialized programs, supporting logical conversion.
   Compiler evaluates assignments and relations relative

     &&  ||  !  =  !=  >  <  >=  <=

with type "char".
  Traditionally, in logical variables, returned not containing integer internal
symbols, specially defined in the use of type declaration typedef in type TINY for
BOOL. The first definition contains logical objects, and the second - integer.
Definition indicates the type, mentioned in the file standard library "types.h".

   Since here several examples of use of logical operations, the following
section provides specific examples of application!.

   1). Type conversion and type casting

   When performing logical arithmetic operations, special attention should be paid to
type conversion and type casting. For example, in the following execution of arithmetic

        int i;
        TINY x,y;

        x = 10; y = 30;
        i = x * y;

   Although the correct value is 300, the variable "i" contains an incorrect value
of 44. This happens as a result of the fact that in the given execution multiplication
is performed according to the rules of logical arithmetic, returning only the lower byte
of the result. In order to obtain the correct result (the number 300), you need to convert
to integer type, or use type casting, as done in the following example:

        i = (int)x * (int)y;

   Alternatively, you can use the following operation:

        i = x << 8 | y;

   When executing the shift operation, both operands of the logical operation shift,
which are transferred for the next processing. But here you also need to specify casting.
The value "y" must be converted to an unsigned type for "i" in the same way as in the
multiplication operation:

        i = (unsigned)x << 8 | y;

   2). With type TINY (char) arithmetic operations return only numbers. The variable of type "char" is used
for numbers. Therefore, before any arithmetic operation, where the result of the operation is a standard
integer divisor. Therefore, if you execute the following program, the variable "i" will be assigned 255:

        int i;
        TINY x,y;

        x = 2;
        y = 3;
        i = x - y;

   In order to obtain the correct result, you must execute both operations in
accordance with type "int":

        i = (int)x - (int)y;

   3). Parameter type matching

   The SOLID C compiler strictly enforces the distinction between types "int" and "char".
Therefore, actual and formal parameters must be clearly consistent with
each other in type. The following example demonstrates an incorrect function call:

func(x)
TINY x;
{
   ...
}
main()
{
   func(1);    /* Incorrect function call */
}

You must make the call as follows:

     func((TINY)1);

   Parameter type matching is not checked at compile time. Therefore,
the following special example shows how to avoid such errors.


                     Register Declaration

   Although the SOLID C compiler provides automatic register allocation of variables,
in a way that allows to receive more effective object code. In the standard
C language, is allowed (keyword "register") only a few variables from all variables
(the total number of variables placed in registers depends on the registers available,
in terms of which operating units will be used). Unlike the standard, the SOLID C
compiler automatically performs optimal allocation of registers, in accordance with the
program flow. At the same time, this can significantly improve the quality of generated code.
However, there are variables that cannot be placed in registers, depending on the level of
optimization. Therefore, the designation "register" differs somewhat from the designation "auto" and
other class of declarations listed below.
   In the following function of the program, the maximum 16 variables of static type standard
placement will be placed in registers, either as variables of type "auto" or "register". To such
type types also include "int", "char", "unsigned" and "pointer".
   Features of the language allow only local variables. The SOLID C compiler
also, of course, supports these features. However, in cases where you have variables
that must be objects of register allocation register, sometimes there are problems. These
are caused by the fact that these "temporary" registers are not accessible.
For example, if you specify

      int n;
      n = 10;
      scanf("%d", &n);
      printf("%d", n);

   If the input is 100, what will the value of the variable "n" be?. Certainly, 100.
The compiler updates the value of the variable from the assignment statement and memory, if
specify the address of this value of this variable. Specifically, problems arise if
in the function call you specify it not in the right place as the parameter by reference.
Consider the example:

      int n;
      int *p;
      p = &n;
      n = 10;
     *p = 100;
      printf("%d", n);

   What should happen, would be that in the right place of the value of the variable "n" will be
assigned to 100, and 10. After these register declarations, between the line "n=10", and
another "printf("%d",n)", we figure out "&n". Therefore the compiler may not update "n" in memory.

                         Additional Information

                 Functions with Fixed Parameters

   In the case of calling functions (functions with fixed parameters), the first
three parameters are passed in registers, and the following ones - on the stack. If any one
of the 3 parameters has type "char", then it is returned in the corresponding 8-bit
register. Parameters (except first) of other types are placed in registers in pairs.
The 4th parameter (if present) and all the following ones are placed on the stack, in the appropriate
location (stack mode ?). At the bottom of the stack are returned two integers.
Parameters are arranged on the stack in groups of two. Values of "char"
parameters are returned in the high byte, and values of the other type are obtained in the
remaining positions.
  In mode compatible with the standard C language, the use of parameters of type
"char" has the following feature: conversion from "char" to type designation and "int" occurs
automatically (the high byte is filled), then after that is mismatching as "other type".

 Type of parameter     First    Second   Third   Fourth    Fifth   ...
_______________________________________________________________________

  char                A        E        C       (SP+2)    (SP+4)  ...

  Other type          HL       DE       BC       (SP+2)    (SP+4)  ...
                                                (SP+3)    (SP+5)  ...
_______________________________________________________________________


                 Functions with Variable Parameters

   When calling a function with variable parameters, all parameters are placed
on the stack (in the appropriate location). The number of parameters placed on the stack is
returned in register pair HL. At the bottom of the stack are returned two integers.
   In "compliance" mode, "char" parameter is returned in the high byte, the value
of the other type is obtained separately.
   In mode compatible with the standard C language, "char" automatically
converts to "int" type (the high byte is filled), then placed
correctly as "other type".
   Parameters of other types are arranged on the stack according to the rules established for

                      Return Values of Functions

   When the function is being called by the program, the function returns its
return value through registers. At the same time, a function of type "char" returns the value
through register "A". Functions of other types return their value through register pair
HL.
  In mode compatible with the standard C language, the returned value is always
returned through register pair HL, except in the case where the function returns
type "char".
  For proper return value and reception from a function that returns different values,
you should use a pointer to the variable, as the parameter through other C-programs.


                       Function Call Sequence

   1. First, the calling program pushes the parameters and returns them to either
registers or placed on the stack according to the rules, described above.
   2. The program calls the function. Typically the word "call" is used.
   3. After the function returns control, the parameters are removed from the stack
(using the "pop" instruction). If the parameters are placed in registers,
using the "pop" instruction is not necessary. It should be noted that in the case
of multiple "pop" commands, the return value (through register "A" or "HL") of the
function does not lose its value.
   4. The value of the function is returned through register "A", if the function returns
type "char". For functions of another type, the value is always returned through
register pair HL.

                  Recommendations on Programming

   1. Avoid using non-recursive functions, if for any reason you are going to use recursion.
Avoiding such situations, use external variables through which parameter passing and temporary
variables that are not assigned to allocate registers when properly efficient object code.

   2. If you specify the compatibility mode with the standard C language (using the "-k" option),
then the amount of object code will be larger, therefore specifying this mode should
use only in cases where compatibility with the standard C language should be higher
than effective object code.

   3. Use type "char" (with 8-bit value range) only for numbers. Arithmetic operations are
performed in logical 8-bit operations more efficiently than with numbers, stored in words (types "int" or "unsigned").
Therefore, variables, having numeric values with a range from 0 to 255,
should be defined as TINY (char).

   4. When using 16-bit variables, you should define as type "unsigned", which in many
situations should be prioritized, before "int".
The reason for this is that in the processor Z80 arithmetic operations are performed correctly
(in particular, shift operations left and right by numbers)
are arranged in their manner.

   5. Definitely set the static mode for external variables.
Variables, defined as external or internal, are not placed into
registers. Therefore, if the goal is to achieve high speed execution of the program, you should
define external variables as "static", and functions as "extern", then the result will be contradictory -
effective object code will be lost. The SOLID C compiler itself allocates registers,
therefore you should clearly define local variables, as "auto".
